You are building a full-stack web application for Company ABC — a chemical manufacturer. The app connects to an existing PostgreSQL database hosted on Neon and allows users to enter, view, and edit operational data including Proforma Invoices, customers, products, outstanding payments, and agents.

---

## TECH STACK

Frontend: React 18 + TypeScript + Vite
Routing: React Router v6
Data fetching: TanStack Query v5 (useQuery, useMutation)
Styling: Tailwind CSS v3
Components: shadcn/ui
Backend: Express.js + TypeScript (thin API layer over Neon)
Database client: postgres (npm: 'postgres') or pg
Deployment: Frontend → Vercel, Backend → Render (or a single Express server serving both)
Environment: .env with DATABASE_URL pointing to Neon

Do NOT use Prisma or any ORM. Write raw SQL queries using tagged template literals (postgres library) or pg.Pool.query(). The schema is already live — never run CREATE TABLE or DROP TABLE. Never ALTER TABLE to remove or rename existing columns.

---

## DATABASE CONNECTION

Use the existing Neon PostgreSQL database. The schema has these tables (already populated with real data):
- lookup_state_codes, lookup_financial_years, lookup_packaging_types (seed/lookup)
- catalog_agents, catalog_products, catalog_product_variants (catalog)
- customers (all trading parties — buyers and suppliers)
- sales_orders, sales_order_lines, sales_pi_sequences (transactional)
- finance_outstanding, finance_outstanding_alerts (finance)

Two useful views already exist:
- v_customer_outstanding — per-customer outstanding summary with total_pending, max_overdue_days, buckets at 30/60/90+ days
- v_sales_orders_summary — orders with buyer name, consignee name, agent name, FY label, line count, total qty

One useful DB function:
- get_next_pi_number(fy_key SMALLINT) — call via SELECT get_next_pi_number($1) to auto-generate the next PI number for a given FY. This is atomic and thread-safe.

## GLOBAL FILTER SYSTEM

This is the most important architectural decision. Every page in the app shares a single global filter context. Filters persist as URL query parameters so links are shareable and the back button works.

### Filter state (React Context + URL params)
```typescript
interface GlobalFilters {
  dateFrom:    string | null   // ISO date, inclusive — applied to order_date
  dateTo:      string | null   // ISO date, inclusive
  fyKey:       number | null   // e.g. 26 for FY 2025-26
  customerId:  string | null   // UUID — filters by buyer_id OR consignee_id
  consigneeId: string | null   // UUID — filters consignee_id specifically
  agentId:     string | null   // UUID
  status:      string | null   // draft|sent|approved|dispatched|invoiced|cancelled
  piFrom:      number | null   // seq_number range start (inclusive)
  piTo:        number | null   // seq_number range end (inclusive)
}
```

### Filter bar component (persistent, appears on every page below the nav)
- Date range: two date pickers (from / to), both inclusive. Selecting a FY preset auto-fills date range to Apr 1 – Mar 31 of that year (or today if current FY).
- FY dropdown: populated from lookup_financial_years, shows fy_label. Selecting FY auto-sets date range.
- Customer dropdown: searchable, populated from customers table. Typing filters the list.
- Consignee dropdown: same customers table, separate field.
- Agent dropdown: populated from catalog_agents.
- Status dropdown: static list ['draft','sent','approved','dispatched','invoiced','cancelled'], multi-select.
- PI # range: two number inputs (from seq_number, to seq_number).
- "N filters active" badge showing count of active non-default filters.
- "Clear all" button resets all filters.
- All filters are optional. No filter = show all data.

### How filters affect every page
Every API call on every page passes all active filters as query parameters. The backend SQL applies them as WHERE clauses:

```sql
SELECT * FROM v_sales_orders_summary
WHERE 1=1
  AND ($1::date IS NULL OR order_date >= $1)
  AND ($2::date IS NULL OR order_date <= $2)
  AND ($3::int  IS NULL OR fy_key = $3)
  AND ($4::uuid IS NULL OR buyer_id = $4 OR consignee_id = $4)
  AND ($5::uuid IS NULL OR consignee_id = $5)
  AND ($6::uuid IS NULL OR agent_id = $6)
  AND ($7::text IS NULL OR status = $7)
  AND ($8::int  IS NULL OR seq_number >= $8)
  AND ($9::int  IS NULL OR seq_number <= $9)
ORDER BY order_date DESC, seq_number DESC
```

The Dashboard KPI numbers MUST update when filters change. Every number on every page is always filter-aware. When a filter is active, show a small "filtered" badge on the relevant KPI card so the user knows they are not seeing the full dataset.

STICKY TOTALS SIDEBAR (updates live as user types):
- Gross value = SUM of all line amounts
- Insurance = gross_value * insurance_pct

- SGST amount = assessable * cgst_rate (same as CGST, shown only if gst_type = CGST_SGST)

- TOTAL = assessable + igst + cgst + sgst + tcs

## KEY BEHAVIOURS AND CONSTRAINTS

1. Never run DDL (CREATE TABLE, DROP TABLE, ALTER TABLE). Read and write data only.

2. The PI number (pi_number) is ALWAYS generated by calling the PostgreSQL function get_next_pi_number(fy_key). Never generate it in JavaScript. Call SELECT get_next_pi_number($1) and use the returned string.

3. When creating a new order: INSERT the order header first (returning order_id), then INSERT each line item referencing that order_id in a loop or batch.

4. The DB trigger fn_cancel_parent_on_revision() handles cancelling the parent PI when a revision is inserted. Do NOT cancel the parent in application code — the DB handles it automatically.

5. GST type auto-calculation: company's home state is Gujarat (state_code = 24). Rule: if buyer_state_code == 24 then gst_type = 'CGST_SGST', else gst_type = 'IGST'. User can override this in the form.

6. Line item packages calculation: Math.ceil(qty_kg / variant.qty_per_pkg). If qty_per_pkg is null or 0, display 0.

7. All monetary calculations happen client-side in real time for live preview (totals sidebar). The server recalculates and stores the final values on save.

8. Outstanding row colour-coding rule: overdue_days >= 90 → red background, >= 60 → orange background, >= 30 → yellow background, else no background colour.

9. The "filtered" badge on KPI cards appears whenever any filter other than the default current FY is active.

10. Filters are AND-ed across different fields. Multi-select within the same field (e.g. multiple statuses) is OR-ed: status IN (...).

11. Date range filter on Dashboard and Orders pages applies to sales_orders.order_date. On the Finance page it applies to finance_outstanding.transaction_date.

12. Export CSV: construct a Blob client-side from the current filtered result set and trigger a browser download. Include all visible columns.

13. The /orders/new route accepts an optional ?buyerId= query param. If present, pre-select and pre-fill the buyer section.

14. Status transitions are one-directional: draft → sent → approved → dispatched → invoiced. Cancelled is terminal except it allows creating a revision. Do not allow backwards transitions in the UI.

## DEPLOYMENT

Frontend → Vercel:
- Root directory: apps/web
- Build command: yarn build
- Output directory: dist
- Environment variable: VITE_API_URL=https://your-render-app.onrender.com

Backend → Render:
- Root directory: apps/api
- Build command: yarn build
- Start command: node dist/index.js
- Environment variable: DATABASE_URL=[your Neon unpooled connection string]

CORS: the Express backend must allow requests from the Vercel frontend domain. Use the cors package and set the origin to the Vercel deployment URL.
